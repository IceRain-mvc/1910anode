# 一、数据库设计

# 1.1、 项目流程：

  • 市场调研->分析需求->创建软件模型图->UI设计->前端设计————————————
  • 成品->测试->安装部署->培训->反馈->创建ER图->数据库模型图->数据库->写java代码

# 1.2、设计数据库与软件项目开发各阶段关系

现实世界>建模>信息世界(E-R图)>规范化>数据库模型图>构建>数据库

# 1.3、软件项目开发周期步骤

  • 1.需求分析阶段： 分析客户的业务和数据处理需求
  • 2.概要设计阶段： 设计数据库的E-R模型图，确认需求信息的正确和完整
  • 3.详细设计阶段： 应用三大范式审核数据库结构
  • 4.代码编写阶段： 物理实现数据库，编码实现应用
  • 5.软件测试阶段： ……
  • 6.安装部署： ……

# 1.4、设计数据库步骤

  • 1.收集信息： *与该系统有关人员进行交流、座谈，充分了解用户需求，理解数据库需要完成的任务
  • 2.标识实体（Entity）： *标识数据库要管理的关键对象或实体，实体一般是名词
  • 3.标识每个实体的属性（Attribute）
  • 4.标识实体之间的关系（Relationship）

# 1.5、实体与实体之间关系（映射基数）

  • 一对一 ：（类似班长和班级的关系），*一个班级有一个班长，同时一个班长也只能管理一个班级。
  • 一对多 ：（类似班长和学生的关系），*多个学生被一个班长管理，同时一个班长可以管理多个学生。
  • 多对多 ：（类似课程和学生的关系），*一个学生可以学多个课程，同时，一个课程可以被多个学生学习。通常情况下借助第三张表体现关系。

# 1.6、实体关系图（E-R图）

  • 矩形：代表实体，一般用名词表示实体
  • 菱形：代表关系，一般用动词表示关系
  • 椭圆：代表属性，一般用名词表示属性

# 1.7、数据库设计范式

  • 第一范式：原子不可再分（表中的每一列不能再拆分成多列）
  • 第二范式：一个表只做一件事（比如：房间表只存房间信息，不存客人信息）
  • 第三范式：依赖主键(表中的每一列必须和主键有直接关系，不能存在间接关系)

# 1.8、数据库的设计原则：

  • 需求>性能>设计（表结构）

# 2.2 权限

# 2.2.1 精度

  • 1、精确到页面/菜单项

  • 2、精确到按钮

  • 3、精确到数据

  • 涉及到的表

    • 小系统（权限少，用户少）

      • 用户表（用户编号，姓名，密码。。。）
      • 权限表（权限编号，权限名称，权限编码，上级编号，权限类型）
        • 1，搜索，search，0，菜单
        • 2，查找，find，1，页面
        • 3, 计数，btnCount,2,按钮
        • 4、取消，btnCancel，2，按钮
      • 用户权限表（编号,用户编号，权限编号）

    • 大系统（权限多，用户多）

      • 用户表（用户编号，姓名，密码。。。）
      • 权限表（权限编号，权限名称，权限编码，上级编号，权限类型）
      • 角色表（角色编号，角色名称，）
      • 用户角色表（编号，用户编号，角色编号）
      • 角色权限表（编号，角色编号，权限编号）
